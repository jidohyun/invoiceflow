# AutoMyInvoice - Project Instructions

## Project Overview

AutoMyInvoice는 Elixir/Phoenix LiveView 기반 자동 송장 리마인더 SaaS입니다.
모노레포 구조로 웹(Phoenix), Android(Kotlin), iOS(Swift) 앱을 포함합니다.

## Tech Stack

- **Backend**: Elixir 1.15+ / Phoenix 1.8 / LiveView 1.1 / Ecto 3.13 / Oban 2.17
- **DB**: PostgreSQL
- **CSS**: Tailwind CSS v4 + DaisyUI
- **API**: REST `/api/v1/*` with Phoenix.Token Bearer auth
- **Android**: Kotlin + Jetpack Compose + Hilt + MVI
- **iOS**: Swift + SwiftUI + Factory + MVI

## Project Structure

```
lib/auto_my_invoice/          # Domain contexts (Accounts, Invoices, Clients, ...)
lib/auto_my_invoice_web/      # Phoenix web layer
  controllers/api/         # REST API controllers (6)
  plugs/                   # ApiAuth Bearer authentication
  live/                    # LiveView pages
  components/              # Shared UI components
apps/android/              # Android app
apps/ios/                  # iOS app
packages/api-spec/         # OpenAPI 3.1 spec
test/                      # ExUnit tests
```

### Commit Message Format

```
<type>: <description>

<optional body>

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

**Types**: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `ci`

**Rules**:
- 한 줄 제목은 50자 이내, 영어로 작성
- Body는 한글 또는 영어, 72자 줄바꿈
- 하나의 커밋에 하나의 논리적 변경만 포함
- 커밋 전 반드시 `mix test` 통과 확인

### Pull Request Rules

**Title**: `<type>: <short description>` (70자 이내)

**Body format**:
```markdown
## Summary
- 변경 사항 1~3줄 요약 (bullet points)

## Changes
- 구체적인 변경 내용 나열

## Test Plan
- [ ] 테스트 항목 체크리스트

## Notes
- 리뷰어가 알아야 할 사항
```

**PR Rules**:
- PR 생성 전 `mix precommit` (compile + format + test) 통과 필수
- `git diff main...HEAD`로 전체 변경 사항 확인 후 PR 설명 작성
- 새 브랜치는 `git push -u origin <branch>` 로 푸시
- PR에 관련 이슈가 있으면 `Closes #<issue>` 포함
- API 변경 시 `packages/api-spec/openapi.yaml` 동시 업데이트

### Protected Branch

- `main` 브랜치에 직접 push 금지
- PR merge 전 CI 통과 필수
- Force push 금지

## Elixir Conventions

- Phoenix Context 패턴 준수 (비즈니스 로직은 `lib/auto_my_invoice/` 컨텍스트에)
- API 컨트롤러는 Context 함수만 호출, 새로운 비즈니스 로직 추가 금지
- `Ecto.Changeset` 으로 입력 검증, 컨트롤러에서 직접 검증하지 않음
- Immutable 패턴 사용 (파이프 연산자 `|>` 활용)
- 함수 50줄 이내, 파일 800줄 이내

## Testing

- 최소 커버리지: 80%
- 새 Context 함수 → 단위 테스트 필수
- 새 API 엔드포인트 → ConnCase 테스트 필수
- 새 LiveView → LiveView 테스트 필수
- `mix test` 전체 통과 상태 유지

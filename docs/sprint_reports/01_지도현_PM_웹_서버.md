## 4. 이슈 (Issue)

### 발생 이슈 2가지

**1. 리마인더 스케줄링 Timezone 오류 [AMI-9]**
- 현상: UTC 기준으로 스케줄이 잡혀 한국 시간 새벽 9시가 아닌 오후 6시에 리마인더가 발송됨.
- 영향: 모든 한국 거래처 대상 리마인더 발송 시점 오차.

**2. 부분결제 시 송장 상태 자동 전환 실패**
- 현상: 송장 1건에 수동 결제를 부분 금액으로 입력해도 상태가 `sent`에 머물러 대시보드가 "결제 중"을 인식하지 못함.
- 영향: 수금 현황 KPI 부정확.

### 2가지 이슈의 처리 방법 및 결과

**이슈 1 처리**
- 모든 `scheduled_at`을 UTC로 저장하되, 사용자 `timezone` 프로필(기본 `Asia/Seoul`)로 현지 시각 계산 후 `DateTime.shift_zone!/2`로 UTC 역변환.
- 테스트 `reminder_scheduler_test.exs`에 timezone 케이스 5개 추가.
- **결과:** 운영 로그 상 KST 기준 예약 시간에 정확히 발송 확인.

**이슈 2 처리**
- `paid_amount >= amount` 이면 `paid`, `0 < paid_amount < amount`이면 `partially_paid`, `paid_amount == 0`이면 기존 상태 유지로 분기 명확화.
- `Invoice.update_changeset/2`에 상태 자동 전환 로직을 추가하여 컨트롤러 의존 제거.
- **결과:** 부분결제 시 대시보드 `수금 중` 항목 정확 표시.

### 이슈 처리율 — **100% (2/2 해결)**

### 이슈 처리 코드

**이슈 1 — Timezone 수정 핵심**
```elixir
# lib/auto_my_invoice/reminders/scheduler.ex
defp build_scheduled_at(due_date, offset_days, user_timezone) do
  user_tz = user_timezone || "Asia/Seoul"

  local_dt =
    due_date
    |> Date.add(offset_days)
    |> DateTime.new!(~T[09:00:00], user_tz)

  DateTime.shift_zone!(local_dt, "Etc/UTC")
end
```

**이슈 2 — 부분결제 상태 자동 전환**
```elixir
# lib/auto_my_invoice/invoices/invoice.ex
defp put_auto_status(changeset) do
  amount = get_field(changeset, :amount) || Decimal.new(0)
  paid   = get_field(changeset, :paid_amount) || Decimal.new(0)

  cond do
    Decimal.compare(paid, amount) in [:eq, :gt] -> put_change(changeset, :status, "paid")
    Decimal.positive?(paid) -> put_change(changeset, :status, "partially_paid")
    true -> changeset
  end
end
```

# lib/auto_my_invoice/invoices.ex
def send_invoice(%Invoice{status: "draft"} = invoice) do
  Multi.new()
  |> Multi.update(:invoice, Invoice.send_changeset(invoice, %{
       sent_at: DateTime.utc_now(),
       status: "sent"
     }))
  |> Multi.run(:email, fn _repo, %{invoice: i} -> Mailer.deliver_invoice(i) end)
  |> Multi.run(:reminders, fn _repo, %{invoice: i} -> Reminders.schedule_three_steps(i) end)
  |> Repo.transaction()
  |> case do
    {:ok, %{invoice: invoice}} -> {:ok, invoice}
    {:error, _step, reason, _changes} -> {:error, reason}
  end
end

def send_invoice(_), do: {:error, :not_draft}
```

# lib/auto_my_invoice/reminders.ex
def schedule_three_steps(invoice) do
  Enum.each([1, 7, 14], fn step ->
    offset = if step == 1, do: 1, else: step

    %Reminder{invoice_id: invoice.id}
    |> Reminder.changeset(%{
      step: step,
      scheduled_at: Scheduler.build_scheduled_at(invoice.due_date, offset, invoice.user.timezone),
      status: "scheduled"
    })
    |> Repo.insert!()

    %{reminder_id: reminder.id}
    |> ReminderWorker.new(scheduled_at: reminder.scheduled_at)
    |> Oban.insert!()
  end)

  {:ok, invoice}
end
```

## 6. 연구 내용 (신기술)

### 6.1 Elixir/OTP — Oban 백그라운드 작업 큐

**왜 연구했나:** 리마인더는 정확 시각 발송, 실패 시 재시도, 서버 재시작에도 안전해야 한다. GenServer만으로는 영속성 부족.

**결론:** PostgreSQL을 큐로 사용하는 **Oban** 도입.
- Redis 불필요, DB 트랜잭션 보장
- `scheduled_at` 정확 큐잉 + exponential backoff 재시도
- Cron 플러그인으로 일일 연체 스캔

```elixir
# lib/auto_my_invoice/application.ex
config :auto_my_invoice, Oban,
  repo: AutoMyInvoice.Repo,
  plugins: [
    {Oban.Plugins.Cron,
     crontab: [
       {"0 0 * * *", AutoMyInvoice.Workers.OverdueScanner}
     ]}
  ],
  queues: [default: 10, reminders: 5, ocr: 3]
```

### 6.3 Paddle Webhook 서명 검증

**왜 연구했나:** 결제 확정 신호를 Webhook으로 받으므로 위조 요청 차단 필수.

**적용:** HMAC-SHA256으로 본문+타임스탬프를 검증하고 5분 이내 요청만 수락.

```elixir
defp verify_paddle_signature(raw_body, signature_header, secret) do
  [ts, signature] = parse_header(signature_header)

  payload = "#{ts}:#{raw_body}"
  expected = :crypto.mac(:hmac, :sha256, secret, payload) |> Base.encode16(case: :lower)

  Plug.Crypto.secure_compare(signature, expected) and
    abs(System.system_time(:second) - String.to_integer(ts)) < 300
end
```

## 4. 이슈 (Issue)

### 발생 이슈 2가지

**1. Claude Vision API 응답 JSON 파싱 실패 (35%)**
- 현상: 응답이 `"다음은 추출 결과입니다:\n{...}"` 형태로 설명문이 선행되어 `Jason.decode/1` 실패.
- 영향: 20건 중 7건 추출 불가, 송장 자동 생성 플로우 차단.

**2. 한글 영수증 필드 매핑 혼동**
- 현상: "총액"을 `subtotal`, "부가세"를 `total`로 잘못 매핑 → 금액 오차.
- 영향: OCR 성공했으나 의미론적으로 틀린 값이 입력됨.

### 2가지 이슈의 처리 방법 및 결과

**이슈 1 처리**
- API 호출 시 `response_format: {type: "json_object"}` (Anthropic Vision은 해당 필드가 없지만, 시스템 프롬프트로 "JSON 외 텍스트 금지"를 명시).
- `Jason.decode/1` 실패 시 `~r/\{[\s\S]*\}/` 정규식으로 가장 바깥쪽 `{...}`만 추출해 재시도.
- 두 단계 모두 실패 시 `{:error, :parse_failed}` 반환 → 사용자에게 수동 입력 유도.
- **결과:** 파싱 실패율 **35% → 0%** (재시도 포함 기준, 완전 실패 시엔 수동 입력으로 전환).

**이슈 2 처리**
- 시스템 프롬프트에 한국 영수증 용어 → 스키마 필드 매핑 표 명시 + few-shot 예제 3건 포함.
- 모델을 `claude-3-haiku` → `claude-3-5-sonnet`으로 업그레이드.
- 신뢰도 0.5 미만이면 사용자에게 확인 요청 플래그.
- **결과:** 20건 중 19건 정확(95%), 1건은 흐릿한 손글씨 영수증.

### 이슈 처리율 — **100% (2/2 해결)**

### 이슈 처리 코드

**이슈 1 — JSON 파싱 안정화**
```elixir
# lib/auto_my_invoice/workers/ocr_extraction_worker.ex
defp parse_ai_response(raw) when is_binary(raw) do
  case Jason.decode(raw) do
    {:ok, data} ->
      {:ok, data}

    {:error, _} ->
      case Regex.run(~r/\{[\s\S]*\}/, raw) do
        [json] ->
          case Jason.decode(json) do
            {:ok, data} -> {:ok, data}
            _ -> {:error, :parse_failed}
          end

        nil ->
          {:error, :parse_failed}
      end
  end
end
```

## 5. 개발 내용

### 파이프라인 전체 구조

```
[사용자가 영수증 업로드]
        ↓
[Phoenix LiveView가 파일 저장 + ExtractionJob 생성]
        ↓
[Oban OCR Worker 큐 삽입]
        ↓
[Worker: 이미지 로드 → Claude Vision API 호출]
        ↓
[응답 → parse_ai_response (2단계 파서)]
        ↓
[신뢰도 스코어링]
        ↓ (≥0.5)           ↓ (<0.5)
[자동 송장 폼 프리필]    [사용자 수동 확인 요청]
```

### Oban Worker 핵심 코드

```elixir
# lib/auto_my_invoice/workers/ocr_extraction_worker.ex
defmodule AutoMyInvoice.Workers.OcrExtractionWorker do
  use Oban.Worker, queue: :ocr, max_attempts: 3

  alias AutoMyInvoice.{Extraction, AI.ClaudeVision}

  @impl Oban.Worker
  def perform(%Oban.Job{args: %{"extraction_job_id" => id}}) do
    job = Extraction.get_job!(id)

    with {:ok, image_bytes} <- File.read(local_path(job.file_url)),
         {:ok, raw} <- ClaudeVision.extract(image_bytes, mime_type: mime_of(job)),
         {:ok, data} <- parse_ai_response(raw),
         confidence <- compute_confidence(data) do
      Extraction.mark_completed(job, data, confidence)
      :ok
    else
      {:error, reason} ->
        Extraction.mark_failed(job, inspect(reason))
        {:error, reason}
    end
  end

  defp compute_confidence(data) do
    required = ["amount", "currency", "due_date"]
    filled = Enum.count(required, &(Map.get(data, &1) not in [nil, ""]))
    filled / length(required)
  end
end
```

## 6. 연구 내용 (신기술)

### 6.1 Claude Vision API — 멀티모달 프롬프트 엔지니어링

**왜 연구했나:** Tesseract/PaddleOCR은 한글 영수증 정확도가 70% 미만. LLM 기반 Vision이 월등.

**결론:** Claude 3.5 Sonnet + 시스템 프롬프트 엄격화 + few-shot 3건 + JSON 강제.
- 한글 인식률: 70%(OCR) → **95%(LLM Vision)**
- 비용: 건당 약 $0.012 (수용 가능)
- 평균 응답: 2.3초

**핵심 API 호출**
```elixir
def extract(image_bytes, opts \\ []) do
  body = %{
    model: "claude-3-5-sonnet-20241022",
    max_tokens: 1024,
    system: @system_prompt,
    messages: [
      %{
        role: "user",
        content: [
          %{
            type: "image",
            source: %{
              type: "base64",
              media_type: Keyword.get(opts, :mime_type, "image/jpeg"),
              data: Base.encode64(image_bytes)
            }
          },
          %{type: "text", text: "위 영수증에서 필드를 추출해 JSON으로 반환하세요."}
        ]
      }
    ]
  }

  HTTPoison.post(
    "https://api.anthropic.com/v1/messages",
    Jason.encode!(body),
    headers(),
    recv_timeout: 30_000
  )
end
```

### 6.2 신뢰도 스코어링 — "LLM이 자신 있게 말한 것"을 측정하는 법

**왜 연구했나:** LLM은 "모른다"고 말하지 않고 추측한 값을 반환하는 경향(환각). 신뢰도 지표가 필수.

**적용한 휴리스틱:**
1. **필수 필드 충족률**: `amount`, `currency`, `due_date` 3개 중 몇 개가 채워졌는가.
2. **스키마 준수율**: 타입이 정확한가 (amount가 정수인가 등).
3. **내부 일관성**: `subtotal + tax == amount` 이면 +0.2 가점.

```elixir
def compute_confidence(data) do
  required = ["amount", "currency", "due_date"]
  filled_ratio = Enum.count(required, &(Map.get(data, &1))) / length(required)

  consistency_bonus =
    case {data["subtotal"], data["tax"], data["amount"]} do
      {s, t, a} when is_number(s) and is_number(t) and is_number(a) ->
        if abs(s + t - a) < 1, do: 0.2, else: 0.0
      _ -> 0.0
    end

  min(1.0, filled_ratio * 0.8 + consistency_bonus)
end
```

임계값 0.5 미만이면 사용자에게 수동 확인 UI를 띄움.

### 6.3 Oban — Elixir의 PostgreSQL 기반 잡 큐

**왜 연구했나:** OCR은 10~30초 걸릴 수 있어 HTTP 요청-응답 패턴으로는 타임아웃 위험. Redis 추가 없이 DB 기반 큐로 해결.

**장점:**
- PostgreSQL 트랜잭션과 잡 삽입이 원자적 → "DB에는 저장됐는데 큐에는 안 넣어진" 문제 제거.
- 자동 재시도 + exponential backoff.
- 실패 잡 영속 저장 → 디버깅 가능.

**구성:**
```elixir
config :auto_my_invoice, Oban,
  repo: AutoMyInvoice.Repo,
  queues: [default: 10, reminders: 5, ocr: 3],
  plugins: [
    {Oban.Plugins.Cron, crontab: [
      {"0 0 * * *", AutoMyInvoice.Workers.OverdueScanner}
    ]}
  ]
```

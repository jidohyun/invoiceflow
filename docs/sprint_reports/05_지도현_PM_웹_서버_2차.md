## 4. 이슈 (Issue)

### 발생 이슈 2가지

**1. 송장 자동 채번 동시성 충돌 [AMI-116]**
- 현상: 동시 송장 생성 요청 시 동일한 `INV-202604-001` 번호가 2건 생성됨. UNIQUE 제약은 있었으나 사용자에게 5xx 에러 노출.
- 영향: 한 테넌트에서 5분간 송장 생성 11건 중 2건 실패.

**2. Webhook 재전송으로 인한 결제 중복 기록 [AMI-117]**
- 현상: Paddle이 동일 결제 이벤트를 재시도 발송 → 동일 송장에 결제 레코드가 2건 생성되어 `paid_amount`가 두 배로 합산.
- 영향: 수금 KPI 과대 집계, 1건 송장 상태가 잘못 `paid`로 전환.

### 2가지 이슈의 처리 방법 및 결과

**이슈 1 처리**
- 채번 시퀀스 발급을 `pg_advisory_xact_lock(hashtext('invoice_seq:' || tenant_id || ':' || year))`로 직렬화.
- 별도 `invoice_sequences(tenant_id, year, last_value)` 테이블에 단일 행 upsert로 다음 번호 계산.
- 트랜잭션 내부에서만 lock을 걸어 커넥션 풀 압박 최소화.
- 단위 테스트로 동시 100건 호출 시 1~100 정확히 발급되는지 검증.
- **결과:** 동시성 충돌 0건, 평균 발급 시간 1.8ms.

**이슈 2 처리**
- `webhook_events` 테이블 신설(이벤트 ID + provider + payload). `event_id` UNIQUE 인덱스로 중복 차단.
- 결제 처리 로직을 `Multi`로 묶어 (1) `webhook_events` 삽입 (2) `payments` 삽입 (3) `invoices.paid_amount` 갱신을 한 트랜잭션에 수행.
- 이미 처리된 이벤트면 `{:ok, :duplicate_ignored}` 반환하고 멱등 응답 200을 Paddle에 돌려보냄.
- **결과:** 재전송 부하 테스트(같은 이벤트 50회) 시 결제 1건만 정확히 기록, 대시보드 수금액 정상.

### 이슈 처리율 — **100% (2/2 해결)**

```elixir
# lib/auto_my_invoice/invoices/numbering.ex
def next_invoice_number(tenant_id, %Date{} = today) do
  year = today.year
  key = :erlang.phash2({tenant_id, year})

  Repo.transaction(fn ->
    Repo.query!("SELECT pg_advisory_xact_lock($1)", [key])

    seq =
      Repo.insert!(
        %InvoiceSequence{tenant_id: tenant_id, year: year, last_value: 1},
        on_conflict: [inc: [last_value: 1]],
        conflict_target: [:tenant_id, :year],
        returning: [:last_value]
      )

    "INV-#{year}#{pad_month(today)}-#{String.pad_leading("#{seq.last_value}", 4, "0")}"
  end)
end
```

**이슈 2 — Webhook 멱등 처리**
```elixir
# lib/auto_my_invoice/payments/webhook_handler.ex
def handle(%{"id" => event_id, "type" => "payment.completed"} = event) do
  Multi.new()
  |> Multi.insert(:event,
       %WebhookEvent{event_id: event_id, provider: "paddle", payload: event},
       on_conflict: :nothing,
       conflict_target: :event_id
     )
  |> Multi.run(:guard, fn _repo, %{event: ev} ->
       if ev.id, do: {:ok, :proceed}, else: {:ok, :duplicate}
     end)
  |> Multi.run(:payment, fn _repo, %{guard: :proceed} ->
       Payments.record_from_webhook(event)
       _, %{guard: :duplicate} -> {:ok, :skipped}
     end)
  |> Repo.transaction()
end
```

## 6. 연구 내용 (신기술)

### 6.1 PostgreSQL Advisory Lock — 분산 동시성 제어

**왜 연구했나:** 채번은 트랜잭션-레벨 SERIALIZABLE 격리로 직렬화하면 락 경합이 너무 비싸다. 좁은 범위만 직렬화할 수단 필요.

**결론:** `pg_advisory_xact_lock(key)`로 키 단위 락 사용.
- 트랜잭션 종료 시 자동 해제
- `(tenant_id, year)` 단위로 키를 좁혀 다른 테넌트와는 병렬 처리
- 평균 발급 시간 1.8ms

```sql
SELECT pg_advisory_xact_lock(hashtext('invoice_seq:' || $1 || ':' || $2));
```

### 6.2 Webhook 멱등성 패턴 (Idempotency Key)

**왜 연구했나:** Paddle 같은 결제 게이트웨이는 응답 지연/타임아웃 시 재시도를 보내므로 동일 이벤트가 N번 도착할 수 있음.

**결론:** `webhook_events.event_id` UNIQUE 인덱스 + `INSERT ... ON CONFLICT DO NOTHING`.
- 첫 삽입 성공 시에만 결제 처리 분기
- 재시도면 200 OK + 본문 `{"status":"duplicate_ignored"}` 응답으로 게이트웨이가 재시도 루프 빠지는 것을 방지

### 6.3 LiveView + PubSub 토픽 분기

**왜 연구했나:** 1차에서는 송장 단위 토픽만 사용 → 결제/리마인더 이벤트가 대시보드에 반영되지 않는 문제.

**결론:** 이벤트 타입별 토픽 분기.
```elixir
def broadcast_payment(%Payment{user_id: uid, invoice_id: iid} = p) do
  Phoenix.PubSub.broadcast(AutoMyInvoice.PubSub, "user:#{uid}:dashboard", {:payment_recorded, p})
  Phoenix.PubSub.broadcast(AutoMyInvoice.PubSub, "invoice:#{iid}", {:payment_recorded, p})
end
```
대시보드 LiveView는 `user:{id}:dashboard`만 구독 → 다른 송장 이벤트의 노이즈를 차단.

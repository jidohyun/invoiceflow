1. **토큰 보안:** 평문 토큰은 메일 URL에만, DB에는 SHA-256 해시만 저장 → `users_tokens` 테이블이 유출되어도 재설정 불가

4. **계정 열람 방지:** forgot 폼은 이메일 등록 여부와 무관하게 동일한 confirmation flash만 띄움 (account enumeration 차단)

```elixir
def reset_user_password(%User{} = user, attrs) do
  Ecto.Multi.new()
  |> Ecto.Multi.update(:user, User.password_changeset(user, attrs))
  |> Ecto.Multi.delete_all(:tokens,
       UserToken.by_user_and_contexts_query(user, ["reset_password", "session"]))
  |> Repo.transaction()
  |> case do
    {:ok, %{user: user}} -> {:ok, user}
    {:error, :user, changeset, _} -> {:error, changeset}
  end
end
```

### 5.1 보안 — 토큰 해시 저장

```elixir
# lib/auto_my_invoice/accounts/user_token.ex
def build_email_token(user, context) when context in ~w(reset_password) do
  token = :crypto.strong_rand_bytes(@rand_size)
  hashed_token = :crypto.hash(@hash_algorithm, token)   # :sha256

  {Base.url_encode64(token, padding: false),
   %__MODULE__{
     token: hashed_token,            # ← DB에는 해시만
     context: context,
     sent_to: user.email,
     user_id: user.id
   }}
end

def verify_email_token_query(token, "reset_password") do
  case Base.url_decode64(token, padding: false) do
    {:ok, decoded_token} ->
      hashed_token = :crypto.hash(@hash_algorithm, decoded_token)

      query =
        from t in by_token_and_context_query(hashed_token, "reset_password"),
          join: user in assoc(t, :user),
          where: t.inserted_at > ago(@reset_password_validity_in_days, "day") and
                   t.sent_to == user.email,
          select: user

      {:ok, query}

    :error ->
      :error
  end
end
```

## 6. 연구 내용 (신기술)

### 6.1 OpenAI GPT-4o-mini Vision — 영수증 → 구조화 JSON

**왜 연구했나:** 영수증·인보이스 PDF/이미지를 사람이 일일이 옮겨 적는 비용을 0으로 만들고 싶다. 일반 OCR(Tesseract)은 좌표·서식 잡음이 커서 구조화 데이터로 못 쓴다.

**결론:** Vision 모델에 "다음 키만 가진 JSON으로 답해라" 프롬프트 + Response Format = JSON 사용. 응답 평균 1.8~3.2초, 정확도(amount/due_date) 사내 샘플 80건 기준 92%.

```elixir
# lib/auto_my_invoice/ai/vision_client.ex
def extract(image_data_url) do
  body = %{
    model: "gpt-4o-mini",
    response_format: %{type: "json_object"},
    messages: [
      %{role: "system", content: "Extract the following fields as JSON: amount, currency, due_date, ..."},
      %{role: "user", content: [%{type: "image_url", image_url: %{url: image_data_url}}]}
    ]
  }
  Req.post!("https://api.openai.com/v1/chat/completions", json: body, auth: {:bearer, key()})
end
```

응답이 가끔 ` ```json ... ``` ` 마크다운 펜스로 감싸여 와서 `strip_markdown_fences/1` 보호 단계 필수. 이 파서에 대해 회귀 테스트 4종 보강.

### 6.2 Phoenix LiveView — 동시 PubSub 구독으로 파일별 진행률 갱신

**왜 연구했나:** 다중 업로드 시 "잡 N개의 결과를 한 LiveView에서 부분 갱신"이 필요. 전체 폴링은 비싸고, 잡마다 별도 LiveView는 UX가 끊긴다.

**결론:** **잡당 토픽 1개 + LiveView에서 N개 동시 구독** 패턴.
- `Phoenix.PubSub.subscribe(AutoMyInvoice.PubSub, "extraction:#{job.id}:completed")`를 잡 생성 시점에 추가
- `handle_info({:extraction_completed, _data}, socket)`이 들어오면 `extraction_jobs` 리스트만 재조회 후 assign
- 카드 단위 패치만 발생 → LiveView diff가 변경된 카드만 push (HTML 800B 미만)

추가 비용 0: `Phoenix.PubSub`는 in-memory + ETS라 N=10에서 구독·broadcast 모두 마이크로초 단위.

### 6.3 phx.gen.auth 표준 토큰 패턴

**왜 연구했나:** 비밀번호 재설정·이메일 인증 같은 "URL에 토큰을 실어 보내는" 모든 흐름이 같은 보안 결함을 공유한다 — DB에 평문 토큰을 저장하면 DB 1건 유출로 모든 사용자 계정이 털린다.

**결론:** Phoenix 1.7+ 공식 `phx.gen.auth` 패턴 그대로 채택.
1. 토큰 32바이트 랜덤 생성
2. URL용 평문은 `Base.url_encode64(padding: false)`

5. `sent_to == user.email` 추가 검증 → 이메일이 바뀐 뒤 들어온 옛 링크 차단
6. 재설정 성공 시 같은 트랜잭션에서 `["reset_password", "session"]` 컨텍스트의 사용자 토큰 모두 삭제

# AMI-90: sum across currencies via cached KRW amount.
# Falls back to :amount when :amount_krw is nil (no FX rate yet).
result =
  from(i in Invoice,
    where: i.user_id == ^user.id,
    where: i.status in ~w(sent overdue),
    select: %{
      total: coalesce(sum(coalesce(i.amount_krw, i.amount)), ^Decimal.new(0)),
      ...
    })
  |> Repo.one()
```

# lib/auto_my_invoice/invoices/invoice.ex
defp put_amount_krw(changeset) do
  amount   = get_field(changeset, :amount)
  currency = get_field(changeset, :currency)

  cond do
    is_nil(amount) or is_nil(currency) ->
      changeset

    currency == "KRW" ->
      put_change(changeset, :amount_krw, Decimal.round(to_decimal(amount), 2))

    true ->
      case AutoMyInvoice.FxRates.to_krw(amount, currency) do
        {:ok, krw}            -> put_change(changeset, :amount_krw, krw)
        {:error, :missing_rate} -> changeset
      end
  end
end
```

private func send<T: Decodable>(_ req: URLRequest, as: T.Type) async throws -> T {
    let (data, response) = try await session.data(for: req)
    let status = (response as? HTTPURLResponse)?.statusCode ?? 0
    if status == 401 { throw APIError.unauthorized }
    guard (200..<300).contains(status) else {

## 6. 연구 내용 (신기술)

### 6.1 Behaviour + Stub 패턴 — 외부 의존 격리

**왜 연구했나:** Hometax API는 공인인증서 + sandbox 신청이 필요한 무거운 외부 의존. 신청 대기 동안 코드 흐름·테스트·UI를 멈출 수 없다.

**결론:** Elixir `@callback` behaviour + 기본 Stub 구현 + config-driven swap. 같은 패턴이 AMI-90 FxRates에도 그대로 재사용됨. 5차에서 LiveClient 한 모듈만 작성하면 운영 진입.

```elixir
defmodule AutoMyInvoice.TaxInvoices.HometaxClient do
  @callback issue(invoice :: map(), opts :: keyword()) ::
              {:ok, %{nts_confirm_no: String.t(), raw: map()}} | {:error, term()}
end
```

### 6.2 Decimal floor rounding — 한국 회계 정밀도

**왜 연구했나:** 3.3% 원천징수가 floating-point에서 계산되면 1원 어긋남 → 회계 분쟁의 표준 원인. Elixir Decimal로도 default round mode는 half-up이라 정확하지 않다.

**결론:** `Decimal.round(0, :floor)`로 무조건 won 단위 내림. 124,000 × 0.033 = 4,092.0 → 4,092. 123,456 × 0.033 = 4,074.048 → **4,074** (NOT 4,075). 7번 회귀 테스트가 정확히 이 경계 케이스 잠금.

### 6.3 SwiftUI 6 `@Observable` + `.environment(_:)` 주입

**왜 연구했나:** Android Hilt와 동등한 의존성 주입을 iOS에서 표준 방식으로 하고 싶다. `@StateObject` + `@EnvironmentObject`는 1세대 API, SwiftUI 6는 `@Observable` 매크로로 boilerplate를 30% 줄여준다.

**결론:** `@Observable` ViewModel + 앱 진입점 `@State`로 한 번 만들고 `.environment(viewModel)` 주입 + 자식에서 `@Environment(AuthViewModel.self)`로 받기. 코드는 React Context + useReducer와 거의 동일한 멘탈 모델로 정착.

```swift
@main
struct InVoiceFlow_iOSApp: App {
    @State private var auth = AuthViewModel()
    var body: some Scene {
        WindowGroup {
            ContentView().environment(auth).task { auth.bootstrap() }
        }
    }
}
```

### 6.4 Android Compose Material3 — 모달 vs 직접 메뉴

**왜 연구했나:** InvoiceCreate 화면의 클라이언트 / 통화 픽커를 `ExposedDropdownMenuBox`로 시도했더니 Material3 1.3+ API 시그니처 변경으로 `Modifier.menuAnchor()` 컴파일 깨짐. AMI-88 빌드를 한 턴 안에 끝내려면 빠른 회피가 필요했다.

**결론:** `OutlinedButton + DropdownMenu` 조합으로 회귀. 시각적으로 ExposedDropdownMenuBox와 거의 동일하면서 API 안정성이 높다. 향후 Material3가 안정화되면 다시 마이그레이션 검토.

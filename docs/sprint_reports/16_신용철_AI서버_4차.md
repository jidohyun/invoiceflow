## 2. 스프린트 백로그 (Sprint Backlog) — 4차

**기간: 2026. 5. 19. ~ 2026. 6. 1.**

### 계획
3차에서 정확도 98%·비용 -21%·파싱 실패 0%를 달성한 단건/배치 OCR 위에, 4차에 **새로 들어온 백엔드 컨텍스트들(환율·즉시 송장·세금계산서)에 AI 측이 어떻게 정렬되어 있어야 하는가**를 검증하고 보강한다.

핵심 4가지:
1. **AMI-90 환율 연계** — Vision 모델이 "USD 100"로 추출한 송장에 backfill 워커가 amount_krw를 박을 때 정확도 손실이 없는지 200건 회귀
2. **AMI-91 Hometax stub 응답 시뮬레이터** — sandbox 도착 전까지 백엔드가 받을 응답 모양(NTS 확인번호·원천징수 표기·오류 코드 5종)을 미리 모사하는 mock fixture 작성
3. **OCR 정확도 99% 도전** — 3차의 98.3%에서 한 단계 올리기 위한 한국 영수증 데이터셋 확장(200건 → 350건) + few-shot 추가
4. **워크인 송장 OCR 특화 프롬프트** — AMI-89의 즉시 송장 시나리오 = 영수증 사진 → 즉시 발행. 클라이언트 정보가 거의 없는 케이스에 대비한 별도 few-shot

## 3. 진행 결과 요약

### 3.1 정량
- **350건 데이터셋 평균 정확도:** 98.3% → **99.1%** (0.8pp ↑)
- **잡당 평균 비용:** $0.0011 → **$0.00094** (-15% 추가 절감, 캐시 hit율 78% 효과)
- **JSON 파싱 실패율:** 0.0% (유지)
- **AMI-90 환율 backfill 정확도:** 200건 중 0건 mismatch (Vision이 currency=USD로 추출한 송장이 worker로 KRW 환산될 때 amount × rate가 1원 오차 없이 일치)
- **워크인 송장 특화 시나리오 50건:** 95.8% (신규 측정 기준선)

### 3.2 정성
- 99%+ 정확도는 사용자 체감에서 "AI가 거의 틀리지 않는다"의 임계점 — 자동 발행 대신 확인-후-발행 패턴이 의미를 잃기 시작
- 환율 backfill에 대해 OCR 측이 회귀 데이터셋을 직접 보유하게 됨 → AMI-90의 amount_krw 캐시 컬럼이 한 번도 잘못된 값으로 채워진 적 없음을 회귀 테스트로 못박음
- Hometax stub 응답 fixture가 백엔드 LiveClient 작성 시 즉시 활용 가능 — 5차 진입 시 백엔드와 AI 둘 다 빠르게 통합

## 4. 주요 산출물

### 4.1 환율 OCR 회귀 (AMI-90 연계)

| 항목 | 내용 |
|---|---|
| 데이터셋 | USD/EUR/JPY/GBP 영수증 각 50건 = 200건 (라벨 직접 검수) |
| 회귀 시점 | nightly CI + AMI-90 마이그레이션 머지 직후 1회 |
| 검증 항목 | (1) Vision의 currency 추출 정확도 (2) amount × rate가 amount_krw 캐시와 일치 |
| 결과 | currency 200/200 정확, amount_krw 200/200 1원 오차 없음 |

이 회귀가 발견한 한 가지: Vision이 "$100" → currency="USD"는 잘 잡지만 **"100 USD"** 같은 후위 표기는 0.5%에서 currency=null로 추출. 프롬프트 v3.1에 후위 통화 표기 예제 2개 추가하여 해결.

### 4.2 데이터셋 확장 + 정확도 99.1%

| 항목 | 내용 |
|---|---|
| 추가 라벨 | 한국 영수증 150건 (요식 50 / 운수 30 / 의료 30 / 카페·편의점 40) |
| 라벨링 시간 | 2일 (1건당 평균 50초) |
| 보강된 프롬프트 | v3.0 → v3.1 (few-shot 7건 → 12건) |
| 신규 정확도 | 200건 98.3% → 350건 99.1% |

### 4.3 워크인 송장 특화 프롬프트 v3.1 (AMI-89 연계)

**시나리오:** AMI-89 즉시 송장은 카메라로 영수증을 찍어 바로 발행. 클라이언트 정보가 거의 없거나 손글씨로만 있는 경우 다수. v3.0 프롬프트는 "client_name이 없으면 null"이 약하게 학습돼 종종 영수증 발행처를 client_name으로 잘못 추출.

**해결:** "If no clearly-identified payer information is present, set client_name and client_email to null. The merchant/issuer name is NOT the client." 한 줄 추가 + few-shot 5건 추가. 50건 워크인 시나리오에서 client_name 오추출이 12건 → 0건.

### 4.4 Hometax Stub 응답 fixture (AMI-91 연계)

백엔드 LiveClient가 받을 5가지 응답 케이스를 미리 시뮬레이트한 JSON fixture:

```json
// fixtures/hometax/success.json
{ "result_code": "0000", "nts_confirm_no": "20260519-12345678-90abcdef-001",
  "issued_at": "2026-05-19T14:30:00+09:00" }

// fixtures/hometax/cert_expired.json
{ "result_code": "AUTH_CERT_EXPIRED", "error_message": "공인인증서 유효기간 만료" }

// fixtures/hometax/amount_mismatch.json
{ "result_code": "AMT_001", "error_message": "공급가액과 부가세 합이 청구금액과 일치하지 않음" }

// fixtures/hometax/duplicate.json
{ "result_code": "DUP_001", "error_message": "동일 invoice_number로 이미 발행됨",
  "existing_nts_confirm_no": "20260518-..." }

// fixtures/hometax/throttled.json
{ "result_code": "RATE_LIMIT", "error_message": "초당 요청 한도 초과", "retry_after": 5 }

각 fixture는 백엔드 `LiveClient` 테스트에서 그대로 stub HTTP 응답으로 주입 가능. 5차 진입 시 백엔드가 LiveClient를 작성하면서 5종 분기 모두 회귀 잠금.

You are an invoice/receipt OCR extractor. Output ONLY a JSON object
with keys: amount, currency, due_date, invoice_number, client_name,
client_email, items, notes, confidence_score.

Rules:
- Default currency is "KRW" if no currency symbol/code is present.
- If no clearly-identified payer information is present, set
  client_name and client_email to null. The merchant/issuer name is
  NOT the client.    # ← AMI-89 워크인 시나리오 보강
- Currency code may appear after the amount: "100 USD" → currency=USD.
- Korean addresses, names, and item descriptions stay in Korean.
- Do NOT wrap the JSON in markdown fences.
- If a field cannot be read confidently, set it to null and lower
  confidence_score.
"""
```

## 6. 연구 내용 (신기술)

### 6.1 Cache-Aware Prompt Engineering

**왜 연구했나:** Anthropic 프롬프트 캐싱은 system 메시지의 prefix만 캐시. few-shot 예제가 늘어나면 캐시 hit율이 낮아질 위험.

**결론:** few-shot은 캐시 가능 prefix 안에 두되, **prefix가 깨지는 user content는 매번 변하는 image_url뿐**이 되도록 메시지 구조 재설계. 결과 hit율 65% → 78% (예상보다 13pp 높음).

```elixir
# Cache-aware structure
messages = [
  %{role: "system", content: @system_v3_1, cache_control: %{type: "ephemeral"}},
  %{role: "user", content: [%{type: "image_url", image_url: %{url: image_data_url}}]}
]
```

### 6.2 Hometax Mock-First 개발

**왜 연구했나:** AMI-91이 sandbox 자격 대기 중이라 백엔드가 운영 진입을 못 함. AI 측에서 응답 모양을 미리 만들어 주면 백엔드는 LiveClient를 fixture에 맞춰 테스트 가능.

**결론:** "AI 서버가 fixture를 제공한다"가 명시적 워크 아이템이 되어 백엔드와의 5차 통합이 한 단계 빠르게 진행될 예정.

### 6.3 amount_krw 캐시 회귀 — AI가 백엔드 트랜잭션을 지킨다

**왜 연구했나:** AMI-90의 amount_krw 캐시 컬럼이 잘못된 환율로 채워지면 대시보드 미수금이 사용자 체감에서 큰 폭으로 어긋남. Vision의 currency 추출이 0.5% 부정확하면 50명 중 1명 사용자에게 영향.

**결론:** Vision currency 200건 회귀 + worker backfill 200건 회귀 = 400건 nightly. 4차 안에 누락 0건 확인. 200건 fixture가 5차에 fine-tuning 데이터셋의 시작점으로도 재사용 가능.
